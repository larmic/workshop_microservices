Kontext

Der erste Booking-Service: läuft auf dem Entwicklerrechner, in der Cloud, auf einem Raspberry Pi — überall gleich. Mit Health-Check (damit Probleme früh auffallen) und externer Konfiguration (damit ein Build in jeder Umgebung läuft).

User Story

Als Betriebsteam möchte ich einen Service, der in beliebigen Umgebungen ohne Code-Änderung läuft und sich automatisch überwachen lässt, damit Probleme früh erkannt werden und Deployment-Pfade nicht spezifisch sein müssen.

Akzeptanzkriterien

  • GET /health liefert 200 wenn der Service gesund ist
  • Logs auf stdout
  • Backend-URLs (Flight/Hotel/Car) konfigurierbar über Umgebungsvariablen
  • GET /info zeigt die aktuelle Konfiguration
  • GET /openapi liefert die OpenAPI-Spec aus — Vorlage: booking/story1/api/openapi.yaml
  • Dockerfile im Service-Verzeichnis baut ein lauffähiges Image (Service lauscht auf Port 8080)
  • Pfad zum Service in services/.env als CUSTOM_BOOKING_PATH eingetragen, sodass make docker-up-hub den Service mitbaut
  • GET /booking/offers aggregiert Flight, Hotel und Car und liefert kombinierte Ergebnisse

ENV:
  FLIGHT_URL=http://flight:8080
  HOTEL_URL=http://hotel:8080
  CAR_URL=http://car:8080

GET /health  → 200 OK
GET /info    → { config: { flightURL, hotelURL, carURL } }
GET /openapi → OpenAPI-Spec (text/yaml)

GET /booking/offers
  → GET FLIGHT_URL/flights
  → GET HOTEL_URL/hotels
  → GET CAR_URL/cars

  // Fehler? Dürfen durchschlagen — kein Error-Handling.

Cloud-Native Booking Service (Story 1)


                    

                

            

        


        


        

            

                

                    Story lesen User Story + Akzeptanzkriterien
                    

                        
Kontext

                        
Statische URLs sind eine schöne Idee für Folien, aber nicht für den Betrieb. Backend-Services registrieren sich bei Consul; der Booking-Service findet sie über logische Namen statt fest verdrahteter Adressen. Fällt eine Instanz aus, übernimmt automatisch eine andere.

                        
User Story

                        
Als Betriebsteam möchte ich Backend-Services über logische Namen finden, statt URLs per Hand zu pflegen, damit Skalierung und Ausfälle ohne nächtliche Config-Edits funktionieren.

                        
Akzeptanzkriterien

                        
    
                            
  • Resolver fragt GET /v1/health/service/{name}?passing=true bei Consul
    • 
                            
  • Statische URLs ersetzt durch logische Namen: flight-service, hotel-service, car-service
    • 
                            
  • Bestehender GET /booking/offers aus Story 1 umgebaut, löst Flight/Hotel/Car über den Consul-Resolver auf statt über statische Env-URLs
    • 
                            
  • POST /booking/bookings orchestriert die drei Backend-Services und liefert eine aggregierte Buchung
    • 
                            
  • Optional: Client-Side Load Balancing bei mehreren Instanzen (zufällige Auswahl)
    • 
                        

                    

                

                

                    Spickzettel Pseudo-Code
                    


resolve(name):
  res = GET consul:8500/v1/health/service/{name}?passing=true
  // res = [ {Service:{Address, Port}}, ... ]
  instance = randomPick(res)
  return "http://{instance.Address}:{instance.Port}"

GET /booking/offers:
  for svc in [flight-service, hotel-service, car-service]:
    url = resolve(svc)
    results[svc] = GET url/...

POST /booking/bookings { flightId, hotelId, carId, customerName }:
  for svc in [flight-service, hotel-service, car-service]:
    url = resolve(svc)
    POST url/bookings
  → { bookingId, customerName, flight, hotel, car }

Service Discovery via Consul (Story 2)

            

                

                    
                    

                        
                        

                    

                

                

                    
                    

                        
                        

                    

                

            

        


        


        

            

                

                    Story lesen User Story + Akzeptanzkriterien
                    

                        
Kontext

                        
Der Flight-Service ist zeitweise nicht erreichbar (Netzprobleme, Überlast, Deployment). Der Booking-Service soll nicht mitfallen, sondern graceful degradieren: Hotel + Car liefern, Flight elegant ausblenden. Ein Circuit Breaker erkennt wiederholte Fehler, schaltet ab und probiert nach einer Pause vorsichtig wieder zu.

                        
User Story

                        
Als Kunde möchte ich auch bei Ausfall des Flugbuchungssystems eine Teilbuchung (Hotel + Mietwagen) durchführen können, damit meine Reiseplanung nicht komplett blockiert wird.

                        
Akzeptanzkriterien

                        
    
                            
  • Circuit Breaker um jeden Backend-Aufruf (Flight, Hotel, Car) — jeweils eigener State
    • 
                            
  • Nach 5 aufeinanderfolgenden Fehlern öffnet der Circuit (OPEN)
    • 
                            
  • Bei OPEN sofortiger Fallback (z. B. flights: [])
    • 
                            
  • Nach 30 Sekunden Wechsel in HALF_OPEN; ein Probe-Call entscheidet, ob CLOSED oder OPEN
    • 
                            
  • Timeout 3 s; Zustandswechsel werden geloggt; Status über Endpoint abfragbar
    • 
                        

                    

                

                

                    Spickzettel Pseudo-Code
                    


state: CLOSED | OPEN | HALF_OPEN
failures = 0
openUntil = 0

call(service):
  if state == OPEN and now < openUntil:
    return fallback()                        // short-circuit
  if state == OPEN and now >= openUntil:
    state = HALF_OPEN                        // probe slot frei

  try service.invoke(timeout = 3s):
    success → failures = 0
              state    = CLOSED
              return result
    failure → failures++
              if failures >= 5:
                state     = OPEN
                openUntil = now + 30s
              return fallback()

Circuit Breaker (Booking Story 3)

            

                

                    
                    

                        
                        

                    

                

            

            

        


        


        

            

                

                    Story lesen User Story + Akzeptanzkriterien
                    

                        
Kontext

                        
Wenn Hotel extrem langsam antwortet, blockieren seine Calls alle Threads des Booking-Service — Flight-Calls verhungern, obwohl Flight gesund ist. Das Bulkhead-Pattern isoliert die Ressourcen: jeder Backend-Service bekommt seinen eigenen Slot-Pool. Voller Pool → sofort 503 ablehnen statt queuen.

                        
User Story

                        
Als Betriebsteam möchte ich dass Probleme mit einem Backend-Service nicht die Aufrufe an andere beeinträchtigen, damit ein langsamer Service nicht das gesamte System blockiert.

                        
Akzeptanzkriterien

                        
    
                            
  • Jeder Backend-Service hat einen eigenen Slot-Pool (Semaphor oder Thread-Pool)
    • 
                            
  • maxConcurrentCalls pro Service konfigurierbar
    • 
                            
  • Bei erschöpftem Pool: sofortige Ablehnung, kein Queueing
    • 
                            
  • Abgelehnte Anfragen liefern 503 Service Unavailable
    • 
                            
  • Metriken über Pool-Auslastung sind verfügbar (in-flight, total, rejected)
    • 
                        

                    

                

                

                    Spickzettel Pseudo-Code
                    


max      = { flight: 5, hotel: 5, car: 5 }   // pro Service
inProgress = { flight: 0, hotel: 0, car: 0 }
rejected = { flight: 0, hotel: 0, car: 0 }

call(service):
  if inProgress[service] >= max[service]:
    rejected[service]++
    return 503                              // sofort ablehnen
  inProgress[service]++
  try:
    return service.invoke()
  finally:
    inProgress[service]--

// Aggregator (Booking) ruft alle drei parallel auf:
//   ein langsamer Hotel füllt nur seinen Pool,
//   Flight + Car liefern weiter normal.

                    Story lesen User Story + Akzeptanzkriterien
                    

                        
Kontext

                        
Eine Reise = Flug + Hotel + Mietwagen. Schlägt die Hotelbuchung fehl, nachdem der Flug schon gebucht wurde, muss der Flug storniert werden. DB-Transaktionen funktionieren nicht über Service-Grenzen — das Saga-Pattern simuliert atomares Verhalten mit expliziten Kompensationsschritten.

                        
User Story

                        
Als Kunde möchte ich eine Komplettbuchung (Flug + Hotel + Mietwagen) durchführen, die entweder vollständig erfolgreich ist oder komplett zurückgerollt wird, damit ich nicht mit einer unvollständigen Buchung dastehe.

                        
Akzeptanzkriterien

                        
    
                            
  • Orchestrierte Saga: Flight → Hotel → Car (zentraler Koordinator)
    • 
                            
  • Jeder Service bietet einen Stornieren-/Kompensieren-Endpoint (DELETE /bookings/{id})
    • 
                            
  • Bei Fehler in einem Schritt: alle vorherigen Schritte kompensieren (Rollback in umgekehrter Reihenfolge)
    • 
                            
  • Status pro Saga abfragbar: PENDING, COMPLETED, COMPENSATING, FAILED
    • 
                            
  • Optional: Retry-Logik für Kompensation; Timeout-Handling; Persistenz für Recovery
    • 
                        

                    

                

                

                    Spickzettel Pseudo-Code
                    


saga = { id, status: PENDING, steps: [] }
booked = []                                    // für Rollback

try:
  flight = POST flight/bookings { ... }
  booked.push({ svc: "flight", id: flight.id })
  saga.steps.push({ svc: "flight", status: BOOKED })

  hotel = POST hotel/bookings { ... }
  booked.push({ svc: "hotel",  id: hotel.id })
  saga.steps.push({ svc: "hotel",  status: BOOKED })

  car = POST car/bookings { ... }
  saga.steps.push({ svc: "car",    status: BOOKED })

  saga.status = COMPLETED

catch error at step X:
  saga.status   = COMPENSATING
  saga.failedAt = X
  for b in reverse(booked):
    DELETE b.svc/bookings/{b.id}              // Kompensation
    mark step COMPENSATED
  saga.status = FAILED

Saga (Booking Story 5)

                
            

            

                

                    
                

                
            

            

            

                Legende
                ✓ BOOKED
                ↶ COMPENSATED
                ✗ KOMP. FEHLER
                ✗ FAILED
                — NICHT AUFGERUFEN
            

            

        

        


        

            

                

                    Story lesen User Story + Akzeptanzkriterien
                    

                        
Kontext

                        
In Story 5 trägt der Booking-Service die volle Verantwortung für die Kompensation: Er ruft synchron DELETE /bookings/{id} gegen jeden Backend-Service auf und behandelt jeden Fehler selbst. Fachlich ist die Stornierung aber Aufgabe der Backends. Lösung: Booking publiziert ein Event, die Backends abonnieren und reagieren eigenständig — wir wechseln von Orchestration zu Choreography.

                        
User Story

                        
Als System möchte ich bei einer fehlgeschlagenen Buchung die Kompensation asynchron an die zuständigen Backend-Services delegieren, damit der Booking-Service nicht für deren Verfügbarkeit haften muss und die Backends ihre eigene Stornierungslogik kapseln können.

                        
Akzeptanzkriterien

                        
    
                            
  • Booking publiziert bei Fehlschlag ein CompensationRequested-Event (statt synchron DELETE)
    • 
                            
  • Flight/Hotel/Car nehmen das Event entgegen, antworten sofort 202 Accepted und kompensieren asynchron
    • 
                            
  • Booking setzt den Step nach Event-Dispatch auf COMPENSATED, die Saga auf FAILED
    • 
                            
  • Idempotenz: eventId wird mitgesendet (persistente Speicherung ist Bonus)
    • 
                            
  • Bonus: Reply-Events, Timeout-Erkennung, STUCK-Status (siehe Recap-Fragen)
    • 
                        

                    

                

                

                    Spickzettel Pseudo-Code
                    


STATE:
  saga   = { id, status: PENDING, steps: [] }
  booked = []                                    // für Rollback

try:
  flight = POST flight/bookings { ... }
  booked.push({ svc: "flight", id: flight.id })
  saga.steps.push({ svc: "flight", status: BOOKED })

  hotel = POST hotel/bookings { ... }
  booked.push({ svc: "hotel",  id: hotel.id })
  saga.steps.push({ svc: "hotel",  status: BOOKED })

  car = POST car/bookings { ... }
  saga.steps.push({ svc: "car",    status: BOOKED })

  saga.status = COMPLETED

catch error at step X:
  saga.status   = COMPENSATING
  saga.failedAt = X
  for b in reverse(booked):
    POST b.svc/events/compensation             // statt DELETE: Event
      { eventId, sagaId, bookingId: b.id }
    mark step COMPENSATED                      // dispatched, kein Reply
  saga.status = FAILED
  // Booking weiß NICHT, ob der fachliche Rollback geklappt hat.
  // Reply, Timeout, STUCK siehe Recap-Frage 1 + 3 (Bonus).

// Backend (flight / hotel / car):
on POST /events/compensation:
  validate(eventId, sagaId, bookingId)
  respond 202 Accepted
  async: rollback(bookingId)

                    Story lesen User Story + Akzeptanzkriterien
                    

                        
Kontext

                        

                            Die Service-Logs zeigen jede Anfrage isoliert: Booking loggt seine Saga, Flight/Hotel/Car loggen
                            ihre lokalen Buchungen — aber niemand kann ein POST /booking/bookings über
                            alle Services hinweg als eine zusammenhängende Operation verfolgen. Korrelation
                            passiert heute nur über Timestamps und Augenmaß. Mit jeder zusätzlichen Komponente wird das
                            schmerzhafter, spätestens bei der Saga aus Story 5/6 (bis zu sechs HTTP-Calls über drei
                            Backends).
                        

                        

                            Eine Trace-ID pro eingehender Anfrage, die durch alle nachgelagerten Aufrufe
                            propagiert wird, löst das Problem — auch ohne eigenes Tracing-Backend: ein
                            grep <trace-id> über die Compose-Logs zeigt den kompletten Ablauf.
                        

                        
User Story

                        

                            Als Entwickler:in im Betrieb möchte ich
                            einen einzelnen Geschäftsvorgang über Service-Grenzen hinweg in den Logs verfolgen können,
                            damit ich Fehlerursachen, Latenzen und Saga-Verläufe gezielt analysieren kann, ohne Logs
                            nach Zeitstempeln zusammenzupuzzeln.
                        

                        
Akzeptanzkriterien

                        
    
                            
  • Bei jedem eingehenden Request am Booking-Service wird ein Trace-Kontext erzeugt oder aus dem
                                Header übernommen (W3C Trace Context: traceparent).
    • 
                            
  • Der Trace-Kontext wird auf jedem ausgehenden HTTP-Call weitergereicht (Flight, Hotel, Car
                                und deren Kompensation).
    • 
                            
  • Auch über die asynchronen Compensation-Events aus Story 6 wird der
                                Trace-Kontext mitgeführt (als Event-Property).
    • 
                            
  • Jede Logzeile aller Services enthält die Trace-ID, sodass
                                docker compose logs | grep <trace-id> den vollständigen Ablauf zeigt
                                — Forward-Buchung und etwaige Kompensation.
    • 
                            
  • Logs sind strukturiert (JSON), damit trace_id ein eigenes Feld ist.
    • 
                            
  • Die OpenAPI-Dokumentation des Booking-Services beschreibt den
                                traceparent-Header.
    • 
                        

                    

                

                

                    Spickzettel Pseudo-Code
                    


// W3C Trace Context: 00-<trace-id-32hex>-<span-id-16hex>-<flags-2hex>

// Server-Middleware: eingehenden Header übernehmen oder neu erzeugen
on incomingRequest(req):
  tc = parse(req.header("traceparent")) ?: generateNew()
  res.header("traceparent") = tc.toHeader()    // zurück zum Client
  ctx = ctx.with(tc)
  next(req.withContext(ctx))

// Client-Inject: pro ausgehendem Hop neue Span-ID, gleiche Trace-ID
on outgoingRequest(ctx, req):
  tc = ctx.get(TraceContext)
  hop = tc.copy(spanId = randomSpanId())
  req.setHeader("traceparent", hop.toHeader())

// Logging: jede Zeile bekommt trace_id als Feld
log.info("forward step done",
  trace_id: ctx.tc.traceId, span_id: ctx.tc.spanId, step: "flight")

// Async-Grenze: Trace-ID als Event-Property mitschicken
event = {
  eventId, sagaId, bookingId,
  traceparent: ctx.tc.toHeader()    // damit der Konsument den Trace fortführt
}
publish(event)

on receiveCompensationEvent(event):
  tc = parse(event.traceparent) ?: generateNew()
  asyncCtx = bgCtx.with(tc)
  go process(asyncCtx, event)       // Logs der Goroutine tragen die trace_id
